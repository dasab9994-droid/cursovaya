# Shop App — клиент-серверное приложение управления магазином

Java + JavaFX + MySQL, архитектура MVC, согласно требованиям задания.

## Возможности

- Регистрация и вход по логину/паролю (пароль хешируется, в БД не хранится в открытом виде).
- Проверка сложности пароля регулярным выражением: минимум 8 символов, хотя бы одна цифра,
  одна заглавная буква и один спецсимвол (`com.shop.service.PasswordUtil`).
- Две роли с разным уровнем доступа (`Роли.Уровень_доступа`):
  - **Клиент** — просматривает товары и способы доставки (только чтение), оформляет свои заказы,
    видит и может изменить количество/удалить только свои заказы, может изменить свои
    регистрационные данные и пароль (вкладка «Профиль»).
  - **Администратор** (`Уровень_доступа >= 2`) — полный CRUD по товарам, способам доставки,
    связке «доступные доставки», ролям, заказчикам, видит и редактирует все заказы.
- Singleton-класс подключения к БД (`com.shop.db.DBConnection`), все запросы — параметризованные
  (`PreparedStatement`), что исключает SQL-инъекции.
- Чёткое разделение Model (`com.shop.model`) / DAO как слой доступа к данным (`com.shop.dao`) /
  View (FXML, `src/main/resources/com/shop/view`) / Controller (`com.shop.controller`).

## Перед первым запуском

1. **Настройте подключение к БД.** Откройте `src/main/resources/db.properties` и укажите ваши
   реальные параметры MySQL. Важно: 8080 в адресе phpMyAdmin — это порт **веб-интерфейса**
   phpMyAdmin, а не порт самого MySQL-сервера. Порт MySQL обычно `3306` (стандарт для XAMPP/
   большинства локальных установок) — если у вас иначе, поправьте `db.port`.

2. **Выполните `sql/02_seed_data.sql`** в phpMyAdmin (или через консоль `mysql`) — он создаёт
   2 роли и одного тестового администратора:
   - Логин: `admin`
   - Пароль: `Admin123!`

   Без этого шага вход в приложение в первый раз будет невозможен (а саморегистрация создаёт
   только роль «Клиент» — иначе любой регистрирующийся мог бы получить права администратора).

3. *(Опционально)* выполните `sql/03_integrity_trigger.sql`, если хотите дополнительно
   продемонстрировать обеспечение целостности данных на уровне самой БД (см. комментарий в файле).

## Сборка и запуск

В этой среде нет доступа в интернет, поэтому проект собран здесь только как исходный код —
соберите его сами на машине с доступом к Maven Central (там же, где у вас установлен MySQL):

```bash
cd shop-app

# Быстрый запуск во время разработки (не требует сборки fat-jar):
mvn javafx:run

# Либо сборка исполняемого JAR со всеми зависимостями внутри:
mvn clean package
java -jar target/shop-app.jar
```

Требуется JDK 17+ и Maven. Если при запуске `java -jar` возникает ошибка вида
*"JavaFX runtime components are missing"* — обычно это означает, что в `pom.xml` для
зависимостей `javafx-controls`/`javafx-fxml` нужно явно указать классификатор платформы
(`win`, `linux` или `mac` — в зависимости от вашей ОС), так как JavaFX поставляет
платформо-зависимые native-библиотеки. В большинстве случаев этого не требуется (приложение
уже использует класс `Launcher`, который не наследует `Application` — это стандартный обход
данной проблемы), но если ошибка всё же возникнет, добавьте классификатор и пересоберите проект.

## Архитектурные решения, которые стоит знать при защите задания

- **Зачем две похожие таблицы — `Доступные_доставки` и `Товары_доставки`?**
  `Доступные_доставки` — это каталог админом разрешённых пар «товар + способ доставки»
  (просто справочник правил). `Товары_доставки` — конкретизированная пара, на которую
  ссылаются реальные заказы (`Документы.id_товар_доставка`). Когда клиент оформляет заказ,
  приложение сначала проверяет правило в `Доступные_доставки`, затем находит существующую
  или создаёт новую запись в `Товары_доставки` (метод `ProductDeliveryDAO.findOrCreate`) и
  только потом создаёт заказ. Администратор может управлять обеими таблицами и напрямую через
  вкладку «Доступные доставки».

- **Почему у `Товары_доставки` нет метода Update?**
  Эта таблица — просто конкретизация пары значений, на которую ссылаются заказы. Изменить
  «куда ссылается» можно через удаление и создание новой записи (`findOrCreate`); отдельный
  Update для двух FK-полей не добавляет функциональности и только усложняет код.

- **Хранение пароля.** Пароль хранится не в открытом виде, а как `base64(соль):base64(SHA-256(соль+пароль))`
  (см. `PasswordUtil`). При каждой регистрации/смене пароля генерируется новая случайная соль.

- **Разделение прав в интерфейсе.** Большинство экранов (`ProductsController`,
  `DeliveriesController` и т.д.) при инициализации проверяют `Session.isAdmin()` и скрывают/
  блокируют кнопки изменения данных для обычных пользователей — таблицы остаются доступны для
  просмотра всем. Вкладка «Заказы» дополнительно фильтрует список заказов по
  `id_заказчика`, если пользователь не администратор.
